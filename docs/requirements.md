## 6. セキュリティ要件

### 認証・アクセス制御

| 要件 | 内容 |
|------|------|
| パスワード管理 | Supabase Authを使用・平文保存禁止 |
| セッション管理 | CookieはHttpOnly・Secure・SameSite=Lax設定 |
| Cookie有効期限 | 365日・期限切れは再ログインを要求 |
| RLS設定 | 自分のaccount_idのデータのみ参照可能 |
| APIキー管理 | 秘密キーはサーバー側のみで使用・フロントエンドに露出しない |

### 通信・データ保護

| 要件 | 内容 |
|------|------|
| 通信暗号化 | 全通信をHTTPS/TLS |
| データ保存場所 | Supabaseの東京リージョン(ap-northeast-1)を指定 |
| 写真データ | Supabase Storageに保存・署名付きURLで参照 |
| パスワードリセット | メール認証経由のみ |

### 個人情報保護法対応

| 要件 | 内容 |
|------|------|
| プライバシーポリシー | 公開前までに作成・申し込みフローに設置 |
| 同意取得 | 申し込み時に同意チェックなしで先に進めない |
| 問い合わせ窓口 | 各施設メールアドレスを窓口として設置 |
| データ削除ルール | 詳細は§18参照 |
| 開示・訂正・削除対応 | 詳細は§18参照 |
| 漏洩時の対応 | 詳細は§18参照 |

### 不正利用対策

| 要件 | 内容 |
|------|------|
| 当日重複チェック | 同一account_idの二重チェックインを防止 |
| 支払いステータス確認 | 支払い済みのみチェックイン可能 |
| ログ保存 | 成功・失敗すべてのチェックイン記録を保存 |

## 8. チェックイン判定仕様

### 判定フローの評価順序

```
1. セッション確認(ログインしているか)
2. 当日チェックイン済み確認
3. 申し込み情報の存在確認(NOT_FOUND)
4. 支払いステータス確認(未払い)
5. 有効期限確認(期限切れ)
6. 施設一致確認(FACILITY_MISMATCH)
7. 時間帯確認(TIME_MISMATCH)
→ 全クリアでチェックイン完了
```

複数条件が重なる場合(例: 未払いかつ期限切れ)は上記の順序で先に該当した条件のエラーを表示する。

### 判定ルール

| 項目 | 仕様 |
|------|------|
| 当日の定義 | JST 0:00〜23:59 |
| 重複チェックのキー | account_id × 日付 |
| 期限切れの境界 | 有効期限日当日はOK、翌日から不可 |
| 支払いステータス | 文字列「未払い / 支払い済み」 |
| 祝日判定 | jpholidayパッケージを使用(§17参照) |

### パス種別ごとの利用可能条件

| パス種別 | 利用可能日時 |
|----------|-------------|
| フリー | いつでも可 |
| 平日 | 平日(月〜金)全時間帯 |
| 土日夜 | 平日19:30以降 + 土日祝日全時間帯 |

### application選択ルール

フェーズ1ではフリー・平日・土日夜の3種別は1人1種別のため、application選択で複数候補が発生するケースはない。特定の場所専用パス追加時(フェーズ2)に選択ルールを別途定義する。

### 種別照合ロジック

```
1. 共通化フラグ確認
   → false: location_idと申し込み施設が一致するか
   → true: どの施設でもOK

2. パス種別確認
   → フリー: 無条件でOK
   → 平日: チェックイン日時が平日(月〜金)か
   → 土日夜: 平日19:30以降 or 土日祝日か

3. 祝日判定: jpholidayパッケージを使用(§17参照)
```

## 10. エラー理由コード一覧

| コード | 意味 | 画面表示テキスト |
|--------|------|------------------|
| `SUCCESS` | チェックイン成功 | チェックインが完了しました |
| `NO_SESSION` | 未ログイン | ログインが必要です |
| `ALREADY_CHECKED_IN` | 当日チェックイン済み | 本日はすでにチェックイン済みです |
| `UNPAID` | 支払い未確認 | お支払いが確認できていません。受付へお越しください |
| `EXPIRED` | 有効期限切れ | ご利用期限が過ぎています。受付へお越しください |
| `FACILITY_MISMATCH` | 共通化されていない(施設不一致) | このパスはこの施設ではご利用いただけません。受付へお越しください |
| `TIME_MISMATCH` | 利用時間外 | 現在の時間帯はご利用いただけません。受付へお越しください |
| `NOT_FOUND` | 申し込み情報なし | 申し込み情報が確認できません。受付へお越しください |
| `SYSTEM_ERROR` | システムエラー | システムエラーが発生しました。受付へお越しください |

失敗時はエラーコードをURLパラメータで渡し、共通エラー画面でテキストを切り替える。`NO_SESSION`と`SYSTEM_ERROR`はDB記録なし。それ以外はすべてチェックイン記録DBに追記する。

パスパラメータ形式を採用。

```
https://yourdomain.com/checkin/[location_id]
```

### location_idと施設の対応表

| location_id | 施設 |
|-------------|------|
| saitama | 埼玉/熊谷 |

### ログイン後のリダイレクト

セッション切れでログイン画面に遷移した場合、ログイン完了後に元のチェックインURLへ自動で戻る。


NFCタグには各施設のURLを書き込んで固定設置する。タグ1枚につき1施設のlocation_idに対応する。

## 15. 管理画面の認証とRLS例外設計

### ロール設計

| ユーザー種別 | ロール | データアクセス |
|--------------|--------|----------------|
| 一般ユーザー | user | 自分のデータのみ(RLS適用) |
| スタッフ | staff | 全件閲覧・支払いステータス更新(サーバー側でservice_roleキー使用) |
| 管理者 | admin | 全操作可能(サーバー側でservice_roleキー使用) |

### 認証方式

- Supabase Authを一本化して使用(お客様・スタッフ・管理者すべて同じ仕組み)
- ロールはSupabase AuthのJWTメタデータ(`app_metadata`)に付与して管理
- 管理画面へのアクセスはロールを確認してから許可する

### RLSとservice_roleの使い分け

| キー | 用途 | 使用場所 |
|------|------|----------|
| anonキー | 未ログインユーザーのアクセス | フロントエンド |
| authenticatedキー | ログイン済み一般ユーザー(RLS適用) | フロントエンド |
| service_roleキー | スタッフ・管理者の全件アクセス(RLSバイパス) | サーバー側のみ・フロントエンドに露出禁止 |

service_roleキーはVercelの環境変数に保存し、Next.js API Routesのサーバー側のみで使用する。フロントエンドのコードに記載しない。

## 16. 同時チェックインの競合対策

### 設計方針

二重のチェックで同一account_idの同日チェックインを防止する。

| レイヤー | 内容 |
|----------|------|
| アプリ側(メイン) | チェックイン時にSupabaseのチェックイン記録DBを参照し当日同一account_idの成功記録を確認 |
| DB側(保険) | `account_id × checkin_date_jst`の組み合わせに部分インデックスで一意制約をかける |

### DB側一意制約の詳細

- 対象: ステータスが「成功」の行のみ
- 理由: 失敗記録(期限切れ・施設不一致・時間外など)は同じ人が複数回記録される可能性があるため除外する
- 実装: 部分ユニークインデックスで対応

```sql
CREATE UNIQUE INDEX unique_checkin_per_day
ON checkins (account_id, checkin_date_jst)
WHERE status = '成功';
```

アプリ側のチェックで99%は防げるが、DB側の制約を最初から入れることで予期しない同時リクエストにも対応できる。実装コストはほぼゼロ。

## 18. 個人情報保護法対応(詳細)

### 退会・利用終了の定義

| 項目 | 内容 |
|------|------|
| フェーズ1の退会受付 | メールまたは受付窓口で申し出・スタッフが管理画面から削除 |
| フェーズ2の退会受付 | マイページ内の退会ページからお客様自身で手続き |
| 自動退会 | 最終チェックインまたは最終ログインから2年間利用がない場合に自動退会 |
| 有効期限切れの扱い | アカウントは残してパスのみ失効扱い。再申し込み時はアカウントをそのまま利用可能 |

### データ削除ルール

| タイミング | 内容 |
|------------|------|
| 退会後(手動・自動問わず) | 1年間保持後に削除 |
| 保持理由 | トラブル対応の証跡として残す |

### 開示・訂正・削除の対応フロー

**フェーズ1(当面の運用)**

```
1. お客様からメールで請求を受け付ける
2. 本人確認(登録メールアドレスからの送信であることを確認)
3. スタッフが管理画面で対象データを確認・対応
4. 対応完了をメールで報告
5. 対応期限: 請求受付から2週間以内
```

**フェーズ2(マイページ実装後)**

マイページ内から開示・訂正・削除をお客様自身で手続き可能にする。

### 漏洩時の対応手順

```
1. 漏洩を発見したスタッフが管理者に即時報告
2. 管理者がSupabaseのアクセスキーを即時無効化
3. 影響を受けたお客様へメールで通知
4. 個人情報保護委員会へ報告(漏洩規模によって義務)
5. 原因調査・再発防止策を実施
```
